## 复审结论

`v6` 已作为当前正式设计基线。原 `v4` 复审中发现的 Markdown 安全、上传安全、登录防暴破、点赞反刷、后台授权、session 加固、安全响应头等设计缺陷，已在 `v5` 中补齐，并由 `v6` 明确继承。

`v6` 新增 MCP Server 设计后，本轮复审未发现架构级或阻塞级设计缺陷，但发现了几项 MCP HTTP 与实现细节层面的非阻塞遗漏，已在 `v6` 和 MCP 实施规格中补齐。当前文档整体保持以下关键边界：

- 不引入第二套服务端技术栈，继续使用 Go。
- MCP 作为协议适配层，不绕过 `service` 层。
- MCP 写能力受 scope、审计、限流、参数校验控制。
- 不开放任意 SQL、任意文件系统访问、任意 Shell 执行。
- 首版不开放删除类 MCP 工具。
- HTTP MCP 默认无状态，后台 session 不与 MCP 鉴权打通。

当前状态：**设计缺陷已收尾，无阻塞项；本轮非阻塞 MCP 细节已补丁闭环，后续重点转入实现阶段的细化与测试验收。**

## MCP 集成复核

### 技术栈一致性

- `v6` 将当前项目设计为 MCP Server，未要求博客作为 MCP Client。
- MCP Server 与现有后端同属 Go 单体服务，不新增 Python/Node sidecar。
- MCP 能力通过 `internal/mcp/` 做协议适配，业务逻辑继续落在 `internal/service/`。
- MCP HTTP 服务与公开博客服务建议分开监听，避免公开站点路由与 MCP 端点混杂。

结论：**未发现技术栈冲突。**

### 安全防护完整性

`v6` 已覆盖以下 MCP 安全边界：

- `Authorization: Bearer <token>` 仅用于 MCP HTTP 鉴权。
- `mcp_clients.token_hash` 存储哈希，不保存明文 token。
- scope 最小授权，不做权限隐式继承。
- HTTP 传输校验 `Origin`，默认绑定 `127.0.0.1`。
- MCP HTTP 默认无状态，避免与后台 session 耦合。
- MCP 审计记录成功、拒绝、限流和错误。
- 写工具必须复用 `v5` 的 slug、发布、上传、Markdown 安全规则。
- Prompt Injection / Resource Injection 被列为独立防护点。

结论：**安全防护设计到位，无阻塞缺陷。**

## 本轮复审补丁项

以下问题属于实现细节缺口，不改变 `v6` 的总体架构结论，已同步修正文档：

| 编号 | 问题 | 风险 | 处理结果 |
|------|------|------|----------|
| M1 | `Streamable HTTP` 只写了 POST + JSON，未明确 `Accept`、`Content-Type`、notification 返回、协议版本头 | 标准 MCP 客户端兼容性不足 | 已在 `v6` 和 MCP 实施规格中补充 HTTP 头、JSON-RPC 响应语义、`202 Accepted`、`MCP-Protocol-Version` |
| M2 | Bearer Token 模式未说明 OAuth / Protected Resource Metadata 的兼容边界 | 公网或第三方客户端接入时鉴权语义不完整 | 已声明首版为私有预注册 token，公网第三方场景需补 OAuth metadata、`WWW-Authenticate` challenge 与 audience 校验 |
| M3 | 401 / 403 鉴权失败未要求返回 `WWW-Authenticate` 与所需 scope | 客户端无法自动理解认证失败原因 | 已补充 401 和 scope 不足时的 `WWW-Authenticate` 要求 |
| M4 | `cover_image` 仅写“站内上传路径”，未明确禁止外部 URL、绝对路径与路径穿越 | 可能引入 SSRF、任意路径引用或错误资源绑定 | 已补充仅允许 `/uploads/YYYY/MM/{uuid}.{ext}` 格式站内资源 |
| M5 | token 哈希只写“不存明文”，未说明哈希方案和常量时间比较 | token 存储和校验实现可能偏弱 | 已补充推荐 `HMAC-SHA256(server_secret, token)` 或同等强度方案，并使用常量时间比较 |

## 非阻塞实现关注点

以下内容不是设计缺陷，但实现阶段需要明确：

1. MCP SDK 版本需要在实现前锁定，并记录对应支持的传输能力。
2. Token 哈希算法、token 明文展示一次、撤销流程需要在 CLI 任务中落实。
3. Tool 参数 schema 应采用严格白名单，未知字段默认拒绝或忽略。
4. HTTP MCP 暴露到公网前，必须补反向代理配置样例和上线检查项。
5. MCP 审计日志需要避免记录完整文章正文、图片 base64 和 token。
6. `stdio` 模式日志必须写 `stderr`，`stdout` 只能输出 MCP 协议消息。
7. 若未来面向通用第三方 MCP 客户端开放，必须新增 OAuth 模式设计，不得直接复用首版私有 token 方案。
