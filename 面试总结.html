整体结论

这批面试题不是单纯的“前端八股”或“AI 概念题”，而是在考察候选人是否能作为一个能落地业务的 AI 全栈工程师：既要讲清楚 Agent、RAG、多智能体、向量检索、上下文治理，也要能解释数据库、缓存、队列、支付、高并发、K8S、SSE/WebSocket、前端兼容和团队协作。

面试官最关注的不是“用了什么工具”，而是三个结果：系统为什么这么设计、线上出了问题怎么排查、AI 引入后到底带来了多少成本与效率变化。

1 AI Agent / RAG 是主线，几乎每家公司都会追问系统架构、检索流程、模型选型和落地效果。
2 全栈工程能力是门槛，缓存一致性、队列可靠性、事务、高并发和容器部署都被频繁提到。
3 前端基础仍然重要，Vue/React、Canvas、动画、兼容、SSE/WebSocket 是典型追问区。

高频主题归纳

一、AI Agent 与多智能体架构

高频问题包括：Agent 系统分层、LangChain / LangGraph 选型、多 Agent 的必要性、通用 Agent 与垂直业务 Agent 的区别、动态路由、工具调用失败处理、Claude Code 核心循环和上下文治理。

  • 需要能解释 Agent 的基本闭环：输入理解、规划、工具调用、执行反馈、结果生成、状态记录。
  • 多 Agent 不应只回答“职责拆分”，还要说明它解决复杂任务隔离、上下文隔离、专业能力分工、失败降级和并行执行问题。
  • LangGraph 的优势可以从状态机、节点编排、条件边、可观测性、失败恢复和复杂流程控制来说明。
LangChain LangGraph Multi-Agent Tool Calling Claude Code

二、RAG、知识库与检索质量

多家公司追问 RAG 流程、文档切分、向量化、向量库选型、混合检索、Re-rank、查询改写、领域词典、政策文件准确性、检索加速和人工审核。

  • 标准流程：数据采集、清洗、切分、Embedding、入库、召回、重排、上下文组装、生成、引用溯源。
  • 准确性要从召回质量、分块策略、元数据过滤、权限过滤、重排模型、答案引用和人工评测集说明。
  • 检索加速可以讲索引优化、缓存、预计算、分层召回、TopK 控制、异步入库和冷热数据分离。
Embedding Vector DB Hybrid Search Re-rank Query Rewrite

三、AI 成本、效果与工程化治理

多个问题围绕 Token 使用量、AI 成本优化、效率提升百分比、国内外模型对比、不同工具和模型效果差异、AI 编码规范、AI 代码 Review、重复推理优化展开。

  • 成本优化要说清楚数据：请求量、平均 Token、缓存命中率、模型分级路由、上下文压缩、批处理和降级策略。
  • 效果评估要避免空泛，可以准备一套指标：交付周期缩短、缺陷率变化、人工审核时间、回答命中率和用户满意度。
  • AI 代码治理重点是规范提示词、限制修改范围、自动测试、Code Review、静态检查和关键路径人工确认。

四、前端基础与实时交互

前端方向集中在 Vue / React 差异、Vue2 / Vue3 响应式、Canvas 无限画布、CSS3 动画、浏览器兼容、HTTP 缓存、SSE 和 WebSocket。

  • Vue2 响应式要讲 Object.defineProperty、数组方法重写、对象新增属性限制；Vue3 要讲 Proxy、Reflect、WeakMap 依赖收集和编译优化。
  • SSE 与 WebSocket 要从连接模型、双向能力、重连、心跳、降级、消息协议和适用场景比较。
  • Canvas / 无限画布应准备坐标系、缩放平移、图层、脏矩形、虚拟渲染、事件命中测试和性能优化。

五、后端、数据库与高并发

这部分覆盖 NestJS、TypeORM / Prisma、事务、缓存一致性、MySQL 慢查询、分库分表、读写分离、队列可靠性、秒杀、支付回调和积分到账。

  • 数据库问题要准备索引设计、执行计划、慢 SQL 抓取、分页优化、批量写入、锁竞争和 IO 异常排查。
  • 秒杀场景要讲限流、库存预扣、Redis Lua、消息队列削峰、幂等、事务和最终一致性。
  • 队列与任务问题要讲持久化、ACK、重试、幂等键、断点续跑、任务状态机和前端状态感知。

六、项目推进、交付与职业画像

多家公司会问没有设计稿如何推进、如何从 0 到 1 搭项目、如何拆解 50 万体量项目并一个月交付、如何管理团队进度、如何快速上手业务、职业规划和技术路线选择。

  • 回答要体现方法论：需求澄清、风险拆解、原型先行、里程碑、验收标准、技术预研、排期和复盘。
  • 职业规划要和岗位匹配：AI 全栈、应用架构、工程提效、技术负责人或技术加管理路线。
  • “最有挑战的问题”要准备真实案例，用背景、冲突、行动、结果和复盘结构回答。

准备重点

必须准备的 5 张图

  • AI 系统整体架构图：前端、网关、业务服务、Agent 编排、模型、工具、知识库、监控。
  • RAG 数据流图：文档入库、切分、向量化、召回、重排、生成、引用。
  • Agent 执行闭环图：任务理解、计划、工具调用、观察、重试、输出。
  • 实时任务状态图：SSE / WebSocket、任务队列、任务状态机、断点续跑。
  • AI 辅助开发流程图：需求、原型、PRD、编码、测试、Review、发布。

必须量化的 5 类数据

  • 系统规模：用户量、日活、请求量、知识库数据量、向量数量。
  • AI 消耗：Token 使用量、平均单次成本、模型调用次数、缓存命中率。
  • 效果指标：召回准确率、回答满意度、人工审核通过率、问题解决率。
  • 性能指标：接口耗时、检索耗时、任务处理时间、并发能力。
  • 业务收益：交付周期缩短比例、人工节省、成本下降、缺陷减少。

最容易被追问的薄弱点

  • 只说“用了 LangChain”，但讲不清为什么不用自研编排或为什么需要 LangGraph。
  • 只说“做了 RAG”，但讲不清分块、召回、重排、评测和准确性保障。
  • 只说“AI 提效明显”，但没有真实数据或对比口径。
  • 只说“用了队列 / 缓存”，但不能解释任务丢失、重复执行、幂等和一致性。
  • 只会前端实现细节，无法把 AI 产品落地、部署、监控、成本串起来。

高频题答题框架

1. “介绍一下你的 AI 项目架构”

建议按“业务目标 → 用户入口 → 后端服务 → Agent 编排 → 模型与工具 → 知识库 → 观测与成本”来讲。不要一上来堆技术名词，先说明这个系统解决什么问题，再说明为什么这样分层。

2. “为什么需要多 Agent？”

可以回答：单 Agent 适合短链路和明确任务，多 Agent 适合复杂任务拆分、角色隔离、并行处理、上下文隔离、失败降级和更清晰的可观测性。关键是补一句：不是为了“显得高级”，而是在任务复杂度、工具数量和上下文长度超过单 Agent 易维护边界后才引入。

3. “RAG 如何保证准确性？”

从数据源可信、切分策略、元数据过滤、混合检索、Re-rank、引用溯源、人工评测集、线上反馈闭环几个层次回答。政策文件类场景尤其要强调版本、时效、来源和引用位置。

4. “AI 成本怎么优化？”

建议分四层：减少调用、减少 Token、选择更合适的模型、提升缓存和复用。具体可以说意图路由、小模型预判、上下文压缩、摘要缓存、Embedding 缓存、工具结果缓存、模型分级和批处理。

5. “SSE 断线续传怎么做？”

重点是服务端持久化任务状态和消息序号，客户端重连时带上会话 ID、任务 ID 或 Last-Event-ID，服务端从上次确认的位置继续推送。窗口关闭太久则转为查询任务最终状态，避免无限保留流上下文。

6. “没有设计稿怎么推进页面？”

回答要体现主动性：先澄清目标用户和核心流程，快速输出低保真原型，拉业务确认信息架构和字段，再用 AI 生成多版 UI 方案，最后沉淀组件规范、交互状态和验收清单。

复习路线

  1. 第 1 阶段：补 AI 系统主线。整理一个自己项目的 AI 架构故事，覆盖 Agent、RAG、模型选型、工具调用、上下文、成本和效果。
  2. 第 2 阶段：补全栈稳定性。复习事务、缓存一致性、消息队列、幂等、限流、秒杀、支付、K8S、任务恢复和监控。
  3. 第 3 阶段：补前端基础。Vue2 / Vue3、React、Canvas、CSS 动画、兼容、HTTP 缓存、SSE / WebSocket 要能结合项目讲。
  4. 第 4 阶段：准备项目案例。至少准备 3 个故事：AI 落地、复杂问题排查、项目推进或交付风险处理。
  5. 第 5 阶段：准备量化数据。把用户量、调用量、Token、成本、性能、效率提升整理成可复述的数字。

面试表达建议

回答时尽量避免“了解过、用过、研究过”这种停留在表面的表述。更有说服力的表达是：当时的业务问题是什么、为什么选择这个方案、实现时遇到什么坑、如何验证效果、最终数据如何、如果重做会怎么改。

由面试原始问题整理生成，可作为 AI 全栈 / Agent 岗位复习提纲。
