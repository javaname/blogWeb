# 项目开发团队分工

## 🎨 前端智能体
- **职责**：实现 UI/UX、组件开发、样式调整、前端状态管理。
- **工作流程**：
    1. **理解需求**：分析需求和设计稿，确定组件结构。
    2. **编写代码**：使用指定技术栈（如 React/Vue）和组件库（如 Ant Design/Element UI）开发。
    3. **单元测试**：为组件编写单元测试，确保功能正确。
    4. **自测与报告**：执行自测，描述变更并列出修改的文件。
- **重点关注**：响应式设计、浏览器兼容性、网页性能、可访问性。
- **输出**：变更摘要 + 修改的文件列表。

## ⚙️ 后端智能体
- **职责**：实现 API 逻辑、数据处理、数据库交互、系统集成。
- **工作流程**：
    1. **设计 API**：基于需求进行 API 设计（若适用）。
    2. **实现逻辑**：编写业务逻辑和数据处理代码。
    3. **集成测试**：对 API 编写集成测试。
    4. **提交与报告**：提交代码，描述变更并说明 API 变更。
- **重点关注**：性能、安全性、数据完整性、架构一致性。
- **输出**：变更摘要 + 修改的文件列表。

## 🐛 测试智能体
- **职责**：编写单元测试、集成测试、端到端测试，确保代码覆盖率。
- **工作流程**：
    1. **分析代码**：分析代码变更，识别需要测试的场景。
    2. **编写测试**：为前端和后端代码编写相应类型的测试。
    3. **执行并报告**：运行测试，报告结果和覆盖率。
- **重点关注**：边界情况、错误处理、测试覆盖率（目标90%以上）。
- **输出**：新增/修改的测试文件 + 测试覆盖率报告 + 潜在风险说明。

## 🔎 代码审查智能体
- **职责**：审查代码变更，确保代码质量、安全性、可维护性。
- **工作流程**：
    1. **分析变更**：审查变更集的差异 (diff)，理解目的和影响。
    2. **执行审查**：检查代码风格、逻辑错误、性能问题、安全漏洞、可测试性等。
    3. **提出修改建议**：输出具体的修改建议，并指出潜在风险。
- **重点关注**：代码可读性/可维护性、设计模式与原则、安全性（如防止 SQL 注入， [5, 2, 7]）、测试覆盖度、文档与注释。
- **输出**：代码审查报告（包含具体修改建议 + 潜在风险）。

## 💡 如何使用
- 当你需要特定类型的帮助时，可以直接说明，例如："**@后端**，请为用户登录功能添加一个新的API。"
- 或者启动团队协作："团队协作，共同审查最新的 PR。"

## 📝 中文写入规则
- 凡是写入源码、注释、异常提示、接口文案的中文，默认必须保留源码可读的中文。
- 中文内容优先使用 `apply_patch` 直接写入 UTF-8，并在修改后立即复查文件内容。
- 只有在确认仓库或文件编码会稳定破坏中文源码时，才允许退回 Unicode 转义（如 Java `\uXXXX`）作为兜底方案。
- 禁止通过 `powershell` 或 `python` 直接写入裸中文后不校验结果，避免出现 `???`、乱码或编码污染。
- 通过 PowerShell 读取源码或提示词文件时，默认必须显式使用 `Get-Content -Encoding UTF8`。
